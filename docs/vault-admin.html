Administrator guide

Setting up Vault for cowbird

Everything a Vault administrator needs to stand up cowbird for a team or family: the KV v2 engine, the per-user policy, how to assign that policy (and why hands-off auto-enrollment behaves the way it does with userpass), and how to manage userpass users day to day.

Overview

What cowbird needs from Vault

Cowbird has no server of its own — every client talks directly to your Vault. The administrator's job is to give each user a Vault login that resolves to a single, shared, identity-templated policy. Everything else (encryption, sharing, recovery) happens client-side.

📦A KV v2 mount

One Key/Value version-2 secrets engine — conventionally mounted at cowbird — holds every user's items, locked identity, share links, public keys, shared envelopes, and inbox messages.

📜One templated policy

A single policy, cowbird-user-access, uses {{identity.entity.id}} templating so the same rules resolve to each user's own subtree. You write it once and assign it to everyone.

🔑An auth method

userpass, token, or AppRole. With userpass — the common choice — the username doubles as the display name other users see when sharing, so pick human-friendly names.

Trust boundary: Vault path ACLs are not the security boundary for item contents — the per-recipient wrapped key is. Item contents are encrypted on the client before they reach Vault, so even broad read access (as the policy grants on the public-key and shared namespaces) never exposes a secret. The policy exists to scope writes and keep users out of each other's private subtrees, not to keep contents confidential.

Prerequisites

Before you start

  • A running Vault you can administer, reachable from your users' machines (e.g. https://vault.example.com:8200).
  • The vault CLI authenticated with a token that can manage secrets engines, policies, identities, and auth methods. (jq is handy for the entity/group recipes below but not required.)
  • Set your environment once so the commands below work as written:
export VAULT_ADDR="https://vault.example.com:8200"
vault login            # as an admin token

Version note. These instructions were verified against Vault 2.0.0 with the userpass auth method. The one behaviour that bites people — userpass not emitting group membership — is covered in detail under Groups & auto-assignment. If you run a different version or auth method, the policy and KV steps are unchanged; only how you assign the policy may differ.

Step 1

Enable the KV v2 secrets engine

Cowbird keeps everything under a single KV v2 mount. The mount path is configurable in the app's first-run setup screen; cowbird is the conventional name and is assumed throughout this guide and in the reference policy. 

vault secrets enable -path=cowbird -version=2 kv

Storage layout

Once users start using the app, the mount fills in with these paths. You never create these by hand — clients write them — but knowing the shape makes the policy and any troubleshooting legible:

users/<entityID>/items/<itemID>     # a user's own items
users/<entityID>/identity           # locked (encrypted) keypair
users/<entityID>/identity.prev      # transient slot during key rotation
users/<entityID>/links/<shareID>    # durable SharedLink records
users/<entityID>/shares/<shareID>   # a user's outgoing ShareRecords
pubkeys/<entityID>                  # public keys + display names
shared/<ownerID>/<shareID>          # shared item envelopes
inbox/<recipientID>/<msgID>         # transient share/revoke messages

All key names are opaque UUIDs. Vault's list output is not ACL-filtered, so anything in a key name is visible to anyone who can list that prefix. Cowbird therefore puts no human-readable data in key names — entity IDs, share IDs, and message IDs are all opaque. Keep it that way if you script anything against this mount.

Step 2

Write the per-user policy

One policy serves every user. It uses {{identity.entity.id}} templating, so each user's token resolves the same rules against their own subtree. The reference policy is checked into the repository as cowbird-user-access.hcl; keep the live policy and that file in sync. 

vault policy write cowbird-user-access cowbird-user-access.hcl

The policy, block by block

# --- own subtree: items, identity, links, share records ---
path "cowbird/data/users/{{identity.entity.id}}/*" {
  capabilities = ["create", "read", "update", "delete"]
}
path "cowbird/metadata/users/{{identity.entity.id}}/*" {
  capabilities = ["list", "delete"]
}

# --- public-key directory: read/list all, write own ---
path "cowbird/data/pubkeys/*" {
  capabilities = ["read"]
}
path "cowbird/metadata/pubkeys" {
  capabilities = ["list"]
}
# Exact match shadows the data/pubkeys/* glob, so "read" must be
# repeated here or reading one's own entry is denied.
path "cowbird/data/pubkeys/{{identity.entity.id}}" {
  capabilities = ["create", "read", "update"]
}

# --- shared envelopes: read all, owner-managed ---
path "cowbird/data/shared/*" {
  capabilities = ["read"]
}
path "cowbird/data/shared/{{identity.entity.id}}/*" {
  capabilities = ["create", "read", "update", "delete"]
}
path "cowbird/metadata/shared/{{identity.entity.id}}/*" {
  capabilities = ["list", "delete"]
}

# --- inbox: recipient reads/deletes own; senders may only create ---
path "cowbird/data/inbox/{{identity.entity.id}}/*" {
  capabilities = ["read", "delete"]
}
path "cowbird/metadata/inbox/{{identity.entity.id}}/*" {
  capabilities = ["list", "delete"]
}
path "cowbird/data/inbox/+/*" {
  capabilities = ["create"]
}

What it grants

NamespaceThe user themselvesEveryone else
users/<you>/* Full CRUD — items, locked identity, share links, share records No access
pubkeys/* Create, read, update own entry Read and list all (public keys are public)
shared/<you>/* Full CRUD on own shared envelopes Read all — safe because contents are encrypted
inbox/<you>/* Read, list, delete own messages Create only — a sender drops a message but cannot read, list, or overwrite

ACL gotcha if you adapt the policy: Vault does not merge capabilities across matching rules — the most specific matching path wins outright. An exact templated path shadowed by a glob must repeat every capability the glob would have granted. That is exactly why the own-pubkey rule (data/pubkeys/{{identity.entity.id}}) repeats read even though data/pubkeys/* already grants it — without that repeat, a user can read everyone's public key but not their own. The create-only inbox/+/* sender rule relies on the same precedence: a user's own templated inbox rules win for their own inbox, while the wildcard governs everyone else's.

Step 3

Enable an auth method

Cowbird authenticates with userpass, token, or AppRole. userpass is the usual choice for human users and is what the rest of this guide assumes. Enable it once: 

vault auth enable userpass

Note the mount accessor — you'll need it for entity aliases and external groups. Capture it now:

vault auth list -format=json | jq -r '."userpass/".accessor'
# e.g. auth_userpass_1c802641

Identities are created on first login. When a user authenticates for the first time, Vault auto-creates an entity for them and the policy's {{identity.entity.id}} resolves to it. If you need a user's entity to exist before they log in (for example, to add them to a group up front), pre-create the entity and alias it to their userpass login — see Entities & aliases below.

Step 4

Assigning the policy: groups & auto-assignment

Because the policy is identity-templated, you only ever attach one policy — cowbird-user-access — to your users. There are three ways to do it, with an important caveat about hands-off auto-enrollment under userpass.

Option A — Per-user token policies (simplest)

Attach the policy directly when you create or update each userpass user. This is the most explicit approach and the one cowbird's own reference deployment uses.

vault write auth/userpass/users/paul \
    password='…' \
    token_policies="cowbird-user-access"
  • Pro: nothing else to wire up; what you see on the user is what they get.
  • Con: you must remember to set it on every user, and there's no single switch to grant or revoke cowbird access fleet-wide.

Option B — An internal identity group (centralized)

Attach the policy to an internal identity group and make each user's entity a member. Now access is managed in one place: add an entity to grant access, remove it to revoke.

# Create the group once, with the policy attached
vault write identity/group \
    name="cowbird-users" \
    type="internal" \
    policies="cowbird-user-access"

# Add members (this REPLACES the member list and the policy set,
# so always pass the full set you want to end up with)
vault write identity/group/name/cowbird-users \
    policies="cowbird-user-access" \
    member_entity_ids="<eid1>,<eid2>,<eid3>"

To append one member without listing them all by hand, read the current set first:

CUR=$(vault read -format=json identity/group/name/cowbird-users \
      | jq -r '.data.member_entity_ids | join(",")')
vault write identity/group/name/cowbird-users \
    policies="cowbird-user-access" \
    member_entity_ids="${CUR},<new-eid>"
  • Pro: one place to manage access; users themselves carry no policy.
  • Con: membership is manual. userpass does not add users to groups for you (see Option C), so you add each entity yourself — typically at the moment you provision the user.

Option C — External groups / true auto-assignment

Vault's hands-off model is the external identity group: you map a group name reported by the auth backend (a "group alias") to an internal group, and Vault populates membership automatically as users log in.

# How it would look with an auth method that reports groups
vault write identity/group name="cowbird-users-ext" \
    type="external" policies="cowbird-user-access"

vault write identity/group-alias name="<backend-group-name>" \
    mount_accessor="<accessor>" \
    canonical_id="<external-group-id>"

This does not work with userpass. userpass in Vault 2.0.0 emits no group claims, so the group alias never matches at login and members are never added automatically. There is no login-time hook to drive auto-enrollment for userpass. If you need genuinely hands-off enrollment, use an auth method that reports group membership — OIDC or LDAP — and external groups + aliases work as designed. With userpass, use Option A or Option B.

Pragmatic "auto-assignment" for userpass

Since there's no login-time hook, the practical substitute is to do the assignment at provisioning time: a small script that creates the user, pre-creates and aliases their entity, and adds that entity to the group — so the policy is in place before they ever log in.

NAME="paul"
ACCESSOR=$(vault auth list -format=json | jq -r '."userpass/".accessor')

# 1. create the userpass login (no policy here; the group carries it)
vault write auth/userpass/users/$NAME password='…'

# 2. pre-create the entity and alias the login to it
EID=$(vault write -format=json identity/entity name="$NAME" | jq -r '.data.id')
vault write identity/entity-alias name="$NAME" \
    canonical_id="$EID" mount_accessor="$ACCESSOR"

# 3. add the entity to the cowbird group
CUR=$(vault read -format=json identity/group/name/cowbird-users \
      | jq -r '.data.member_entity_ids | join(",")')
vault write identity/group/name/cowbird-users \
    policies="cowbird-user-access" \
    member_entity_ids="${CUR:+$CUR,}$EID"

Entities & aliases

The alias in step 2 above is what binds the userpass login paul to the entity you created, so that when Paul logs in Vault reuses that entity rather than minting a fresh one. Get this right and the entity is a group member from the start; get it wrong and the user logs in under a different, group-less entity and sees no access. To inspect what a login resolves to:

vault read identity/entity/name/paul        # entity, its aliases, group membership
vault list identity/entity/name             # all entities by name

Step 5

Managing userpass users

Day-to-day operations on the userpass auth method. Remember that with userpass the username is the display name other cowbird users see in the share picker, so choose names people will recognise.

Create a user

# Option A style — policy on the user
vault write auth/userpass/users/paul \
    password='…' \
    token_policies="cowbird-user-access"

# Option B style — no policy here; the identity group carries it
vault write auth/userpass/users/paul password='…'

List, read, and update

vault list auth/userpass/users                 # all usernames
vault read auth/userpass/users/paul            # policies, token TTLs, etc.

# Change a user's password (admin reset)
vault write auth/userpass/users/paul/password password='new-…'

# Change which policies a user's tokens get
vault write auth/userpass/users/paul token_policies="cowbird-user-access"

An admin password reset does not recover cowbird data. The Vault login password and the cowbird unlock password are deliberately separate secrets (along with the export passphrase). Resetting the Vault password lets the user log in again, but it cannot decrypt their stored keypair. If a user forgets their unlock password and never exported a recovery key, their data is unrecoverable — by design, there is no operator-side reset. Encourage users to export a recovery key (the app's Export Recovery Key action) and store it safely.

Token lifetime

Cowbird renews the user's Vault token in the background, but the auth method's TTLs still set the ceiling. If sessions expire sooner than you'd like, tune them on the user (or the mount):

vault write auth/userpass/users/paul \
    token_ttl="1h" token_max_ttl="12h"

Disable or offboard a user

Deleting the userpass login revokes the ability to log in:

vault delete auth/userpass/users/paul
vault token revoke -mode=path auth/userpass/      # optional: kill live tokens

Deleting the login does not delete the user's stored data. To fully reclaim their footprint (the key names are opaque UUIDs, so list before deleting):

  • Their subtree under cowbird/metadata/users/<entityID>/… — list each prefix and delete the keys (KV v2 has no recursive delete).
  • Their public-key entry cowbird/pubkeys/<entityID>, so no one can share to them any more.
  • Their outgoing shared envelopes under cowbird/shared/<entityID>/…, which revokes anything they had shared with others.
  • Optionally the identity entity: vault delete identity/entity/name/paul.

Items others shared with this user are wrapped to that user's key and held in the owners' subtrees; only those owners can revoke them, which the cowbird client does for them on removal.

Step 6

Verifying a user can access cowbird

A quick CLI smoke test confirms a freshly provisioned user resolves the policy correctly before they ever open the app. 

# Log in as the user and capture their token
TOKEN=$(vault login -method=userpass -token-only \
        username=paul password='…')

# Confirm the policy is attached (directly or via group)
VAULT_TOKEN="$TOKEN" vault token lookup -format=json \
  | jq '.data.policies'

# Reading the directory should succeed; a write under their own
# subtree should succeed; both confirm the templated rules resolved.
VAULT_TOKEN="$TOKEN" vault kv list cowbird/pubkeys 2>/dev/null || \
  echo "(empty directory is fine on a brand-new mount)"

If the policy list is empty or the directory read is denied, the most common cause is the entity/alias mismatch from Entities & aliases — the user logged in under a different entity than the one you put in the group.

Operational notes

Things to keep in mind

  • ACL rules don't merge. The most specific matching path wins outright; an exact path shadowed by a glob must repeat the glob's capabilities. Re-read the policy notes before editing the policy.
  • Key names are public. Vault list output isn't ACL-filtered. Keep everything in this mount keyed by opaque UUIDs.
  • userpass reports no groups. External-group auto-enrollment won't work; assign per-user or via a manually-maintained internal group. Switch to OIDC/LDAP if you need hands-off enrollment.
  • Three separate secrets. The Vault credential (in the OS keyring), the unlock password (in the user's head), and the export passphrase (in the user's head) are independent. No operator can decrypt a user's data, and no operator reset exists.
  • Keep the policy file in sync. The live policy and cowbird-user-access.hcl in the repo should always match.
